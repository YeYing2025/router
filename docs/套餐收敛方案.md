# 套餐收敛方案

这篇文档定义 Router 后续要收敛到的用户态套餐模型，以及从当前实现迁移过去的最小方案。

它回答的是下面四类问题：

1. 用户最终应该看到什么
2. 后台还保留哪些运行时概念
3. 订阅、购买、升级、降级、转化怎么收口
4. 在不一次性推翻现有实现的前提下，应该先改什么

边界：

- 当前套餐语义与并发说明见 [套餐说明.md](./套餐说明.md)
- 当前请求准入、YYC / 按次扣减见 [扣费逻辑.md](./扣费逻辑.md)
- 当前接口字段口径见 [接口文档.md](./接口文档.md)
- 当前支付与套餐购买链路见 [支付方案.md](./支付方案.md)

## 1. 核心结论

Router 后续应收敛成下面这个用户态模型：

```text
一个用户
= 一个当前主套餐（可无）
+ 一个下期套餐（可无）
+ 一份独立余额
```

其中：

- 用户只感知“套餐”和“余额”
- 用户不感知 `group`
- 用户同一时刻只允许有一个当前生效主套餐
- 用户可以多次充值余额
- 套餐不足时，是否允许自动走余额，由套餐规则决定

一句话：

```text
group 是后台运行边界
套餐和余额才是用户资产边界
```

## 2. 为什么要收敛

当前实现是能跑的，但产品语义和运行时语义混在了一起。

当前状态有三个问题：

1. 用户实际只会理解“我买了哪个套餐”，不会理解“我在不同 group 下各有一个套餐”
2. 前台如果继续围绕 `group` 暴露信息，会持续制造理解成本
3. 后续做升级、降级、转化时，如果允许多份并存订阅，会让购买状态机快速复杂化

因此需要明确拆层：

- 产品层：只讲套餐和余额
- 运行层：继续保留 `group`、模型、渠道、供应商、分组范围

## 3. 目标产品模型

### 3.1 用户资产

用户只看三类状态：

1. 当前套餐
2. 下期套餐
3. 账户余额

这里的“下期套餐”是为了支持：

- 降级
- 到期转化
- 已排期的套餐变更

### 3.2 当前套餐

当前套餐表示“此刻正在生效的用户主权益”。

它可以是：

- `yyc_quota`
- `request_quota`
- 将来其他套餐类型

但无论类型如何，产品上都只允许一个当前主套餐。

### 3.3 账户余额

余额始终独立存在：

- 可以多次充值
- 不因为套餐变更而消失
- 可以作为套餐的兜底

余额不是套餐的附属物，而是独立账本资产。

### 3.4 下期套餐

下期套餐不是“额外多买一个套餐给用户自己选择”，而是“已确定的下一周期权益安排”。

它的目标是解决下面这些动作：

- 当前高级套餐，到期后切低级套餐
- 当前 YYC 套餐，到期后转成按次套餐
- 当前套餐不变，但提前续费排到下一周期

## 4. 后台运行模型

用户不感知 `group`，不代表系统要移除 `group`。

`group` 在后台仍然保留，继续承担这些职责：

1. 定义套餐可用的供应商、模型和渠道范围
2. 决定模型路由、倍率和分组策略
3. 作为运行时查找技术范围的内部字段

因此建议的收口方式不是删除 `group_id`，而是：

```text
保留 package.group_id
隐藏 subscription.group_id 的用户态意义
不再把 group 作为用户购买和展示语义
```

## 5. 订阅模型收敛

## 5.1 当前问题

当前订阅模型更接近：

```text
一个用户
可以按 group 同时存在多份 active subscription
```

这对运行时方便，但对产品不自然。

### 5.2 目标模型

应收敛成：

```text
一个用户
最多一份 current subscription
最多一份 next subscription
```

这里的 `current` / `next` 是用户态语义，不是数据库字段名要求。

### 5.3 最小实现建议

第一阶段不一定立刻重构整张表，可以先通过约束和服务层规则收口：

1. 同一用户同一时刻只允许一份“当前生效主套餐”
2. 同一用户最多一份“待生效主套餐”
3. `group_id` 继续从套餐继承，但不再作为允许多主套餐并存的理由

也就是说：

```text
先收口行为
再决定是否重构表结构
```

## 6. 用户接口应该怎么收口

当前最关键的用户接口是：

`GET /api/v1/public/user/package/subscription`

### 6.1 当前口径问题

当前返回里包含：

- `has_active_subscription`
- `subscription`
- `subscriptions`

这会让前台天然倾向于把它当“订阅列表接口”来用。

### 6.2 目标口径

后续应收口成“当前套餐视图接口”，返回重点应变成：

- `has_active_subscription`
- `current_package`
- `next_package`
- `usage`
- `allow_balance_fallback`

其中：

- `current_package`：当前主套餐
- `next_package`：下一周期套餐，没有则为空
- `usage`：当前套餐的用量或剩余口径

### 6.3 兼容策略

短期内可以保留：

- `subscription`
- `subscriptions`

但应明确：

- `subscription` 仅兼容旧前端
- `subscriptions` 仅兼容旧调用方
- 新前端和新接口文档应改为围绕 `current_package` / `next_package`

## 7. 购买动作状态机

后续套餐购买动作建议固定成有限集合，不让前台自行推断复杂语义。

### 7.1 推荐动作

- `purchase`
- `renew`
- `upgrade`
- `downgrade`
- `convert`

### 7.2 语义定义

#### `purchase`

用户当前没有主套餐时购买。

结果：

- 立即生成当前主套餐

#### `renew`

用户购买与当前套餐相同的套餐，用于延长周期。

结果：

- 当前周期不变
- 在尾部顺延一个新周期

#### `upgrade`

用户切换到更高规格套餐。

建议规则：

- 默认立即生效
- 原套餐剩余价值按规则折算
- 目标套餐生成新的当前主套餐

#### `downgrade`

用户切换到更低规格套餐。

建议规则：

- 默认下一周期生效
- 不建议立即生效

原因：

- 避免立即降级带来的套利和解释成本

#### `convert`

跨套餐类型切换，例如：

- `yyc_quota -> request_quota`
- `request_quota -> yyc_quota`

建议初期规则：

- 默认下一周期生效
- 先不支持“即时跨类型切换 + 剩余权益复杂折算”

## 8. 为什么升级和降级不应同规则处理

升级和降级不是一回事。

### 8.1 升级

升级的典型诉求是：

```text
我现在就不够用了
我要立即获得更高权益
```

因此升级更适合立即生效。

### 8.2 降级

降级的典型诉求是：

```text
我下个周期不需要这么高规格
```

因此降级更适合下一周期生效。

如果强行让降级也支持立即生效，会带来：

- 退款解释问题
- 已消耗权益如何重算的问题
- 用户套利空间

## 9. 当前实现下的最小改造顺序

这里强调顺序，避免同时改数据、接口、前台导致风险扩散。

### 第一步：先收口规则

先把下面几条定死：

1. 单用户同一时刻只允许一个当前主套餐
2. 单用户最多一个下期套餐
3. 升级立即生效
4. 降级默认下期生效
5. 转化默认下期生效

### 第二步：收口服务层行为

优先改这些服务逻辑：

- 分配套餐
- 购买套餐
- 续费
- 升级
- 当前套餐查询

目标是让服务层先不再产生“多主套餐并存”的结果。

### 第三步：调整用户接口

优先改：

- `GET /api/v1/public/user/package/subscription`
- `POST /api/v1/public/user/topup/package/preview`
- 套餐购买订单预览中的 `operation_type` 推断逻辑

### 第四步：调整前台

前台只保留三块主展示：

1. 当前套餐
2. 下期套餐
3. 余额

### 第五步：最后再考虑表结构优化

如果服务层收口后仍然觉得当前表结构不自然，再决定是否：

- 新增更明确的状态字段
- 增加“当前 / 下期”约束
- 历史订阅和未来订阅分表

但这不应成为第一步。

## 10. 迁移原则

为了降低风险，迁移时要遵守下面三条原则。

### 10.1 先兼容，后清理

第一阶段允许：

- 旧字段继续返回
- 旧数据继续存在
- 旧记录继续可读

但新写入规则必须收紧。

### 10.2 先改写路径，后改读路径

优先保证：

- 新购买
- 新续费
- 新升级
- 新降级

不会继续制造新的模型混乱。

再逐步把读取口径切到新模型。

### 10.3 先约束行为，再考虑物理去重

不需要一上来就把历史多订阅全量清理。

更稳的做法是：

1. 先阻止新问题继续产生
2. 再根据历史数据特点决定清理方式

## 11. 暂不建议现在做的事

下面这些事当前不建议优先做：

1. 让用户自己选择 `group`
2. 继续增强“多 active subscriptions”前台展示
3. 立即引入复杂的即时降级退款规则
4. 立即支持复杂的跨类型即时转化
5. 为套餐单独再建一套用户可见的供应商 / 模型选择器

这些都会把产品心智重新拉回运行时层面。

## 12. 一句话版本

后续套餐体系应收敛成：

```text
用户只看一个当前套餐、一个下期套餐和一份余额
group 留在后台做运行时边界
升级立即生效
降级和转化默认下期生效
```

## 13. 现有代码映射

这一节把“方案”映射到当前代码入口，后续实施时直接按这里找。

### 13.1 订阅写入与替换

当前核心入口在：

- `internal/admin/model/user_package_subscription.go`

关键函数：

- `AssignServicePackageToUserWithDB`
- `RenewServicePackageForUserWithDB`
- `UpgradeServicePackageForUserWithDB`

当前行为特点：

1. `AssignServicePackageToUserWithDB` 会创建一条订阅记录
2. 它当前按“套餐作用域身份”替换旧订阅，而不是按“单用户唯一主套餐”替换
3. `RenewServicePackageForUserWithDB` 当前按“同 package/group 续尾部周期”处理
4. `UpgradeServicePackageForUserWithDB` 当前本质上是立即再分配一个新套餐

这意味着：

```text
当前写路径的核心问题不是不能写
而是替换边界仍然偏向 group / package scope，而不是用户主套餐
```

### 13.2 当前套餐读取

当前核心入口在：

- `internal/admin/model/user_package_subscription.go`
- `internal/admin/controller/user/handler.go`

关键函数：

- `GetActiveUserPackageSubscription`
- `GetActiveUserPackageSubscriptionForGroup`
- `ListActiveUserPackageSubscriptions`
- `loadActiveUserPackageSubscriptionPayload`

当前行为特点：

1. 读接口会先取“活跃订阅列表”
2. 再构造：
   - `subscription`
   - `subscriptions`
3. `selectPrimaryActiveUserPackageSubscriptionView` 目前还带有“优先 YYC 套餐”的选择逻辑

这说明当前用户接口仍然在输出“多订阅视角”，不是“当前主套餐视角”。

### 13.3 套餐购买预览与动作推断

当前核心入口在：

- `internal/admin/model/topup_order.go`
- `internal/admin/controller/user/handler.go`

关键函数：

- `resolvePackagePurchaseOperationType`
- `PreviewPackagePurchaseWithDB`

当前行为特点：

1. 预览时先按“目标套餐对应的 package/group”找当前生效订阅
2. 没有活动订阅就推断为 `purchase`
3. 同套餐推断为 `renew`
4. 不同套餐推断为 `upgrade`

当前缺口：

- 没有正式的 `downgrade`
- 没有正式的 `convert`
- “当前套餐是谁”仍然是按目标套餐范围去探测，而不是按用户唯一主套餐去判断

### 13.4 套餐购买订单完成后的兑现

当前核心入口在：

- `internal/admin/model/topup_order.go`

关键分支：

- 订单完成后调用
  - `RenewServicePackageForUserWithDB`
  - `UpgradeServicePackageForUserWithDB`
  - `AssignServicePackageToUserWithDB`

这意味着后续如果要支持：

- `downgrade`
- `convert`
- `next subscription`

最终还是要落回这里增加兑现分支。

## 14. 实施清单

这里给出建议的最小实施顺序，尽量减少返工。

### 阶段 1：服务层收口

目标：

```text
禁止继续产生“单用户多主套餐并存”的新写入
```

优先改这些函数：

1. `AssignServicePackageToUserWithDB`
2. `RenewServicePackageForUserWithDB`
3. `UpgradeServicePackageForUserWithDB`

要达到的效果：

1. 同一用户只允许一个当前主套餐
2. 同一用户最多一个待生效下期套餐
3. 替换逻辑不再以 group 为用户态主边界

### 阶段 2：预览和动作推断收口

优先改这些函数：

1. `resolvePackagePurchaseOperationType`
2. `PreviewPackagePurchaseWithDB`

目标：

1. 改成围绕“用户当前主套餐”推断动作
2. 支持显式区分：
   - `purchase`
   - `renew`
   - `upgrade`
   - `downgrade`
   - `convert`

### 阶段 3：用户接口收口

优先改这些函数：

1. `loadActiveUserPackageSubscriptionPayload`
2. `GetCurrentUserActivePackageSubscription`
3. `GetUserActivePackageSubscription`

目标：

1. 输出 `current_package`
2. 输出 `next_package`
3. 保留兼容字段 `subscription` / `subscriptions`
4. 让前台新页面不再依赖多订阅列表

### 阶段 4：订单兑现收口

优先改：

- `internal/admin/model/topup_order.go` 中套餐订单完成后的兑现分支

目标：

1. `upgrade` 立即生效
2. `downgrade` 生成下期套餐
3. `convert` 生成下期套餐
4. `renew` 只延长当前套餐尾部

### 阶段 5：前台改造

用户前台重点页面：

- `web/src/pages/TopUp/CurrentPackagePage.jsx`
- `web/src/pages/TopUp/PackagePurchasePage.jsx`

目标：

1. 当前套餐
2. 下期套餐
3. 余额兜底说明
4. 不展示 group 语义

## 15. 当前最大的兼容风险

实施时要特别注意下面三个风险点。

### 15.1 历史多订阅数据

如果历史上已经存在：

- 同一用户多个 active subscription

那么新逻辑需要先定义：

- 谁是当前主套餐
- 谁应该被标记为 replaced / pending / historical

这一步不能靠前台猜。

### 15.2 预览和兑现不一致

如果只改了预览，不改订单完成后的兑现逻辑，就会出现：

- 预览告诉用户是降级
- 实际落库仍按升级或新购处理

因此预览与兑现必须成对调整。

### 15.3 读取口径和扣费口径脱节

如果用户接口已经改成“单主套餐视角”，但运行时扣费仍然在按旧的多订阅口径查找，就会出现：

- 用户看到的是 A 套餐
- 实际扣的是 B 套餐

这会比纯展示问题更严重。

因此服务层与读取层要一起收口，不能只改前台接口。
